# L2 Memory Writer Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Add a standalone L2 page where advisors can capture meeting notes/transcript, extract candidate memories/actions, approve them, and write approved memories to Neo4j through the existing backend.

**Architecture:** Keep L1/L1.5 read/query flow intact. Add a separate `/l2/[meetingId]` route and a client-side `L2MemoryWriter` component that reuses existing extract/approve APIs. Use approve-before-write so GPT/voice proposals never automatically pollute Neo4j.

**Tech Stack:** Next.js App Router, React client components, TypeScript, Tailwind CSS, existing Neo4j driver helpers, existing demo extraction, existing `/api/memory/approve` write endpoint.

---

## Current Context / Assumptions

- Repo: `toorcn/IH-Goat`.
- Current branch: `feat/l2-memory-writer` from latest `main`.
- Existing L1/L1.5 routes remain unchanged:
  - `/briefing/[meetingId]`
  - `/meeting/[meetingId]`
  - `/post-meeting/[meetingId]`
- Existing backend pieces to reuse:
  - `POST /api/meetings/[meetingId]/extract`
  - `POST /api/memory/approve`
  - `saveApprovedMemory(memory)` in `lib/neo4j-memory.ts`
- Existing extraction is deterministic demo extraction via `extractMeetingSignals`; this is acceptable for hackathon MVP and safe for demo.
- L2 should be presented as its own page/tab, not an add-on inside L1/L1.5.

---

## Proposed Approach

Create a focused standalone route:

```txt
/l2/meeting-2026-06-20-tan
```

The page loads the meeting context server-side and renders:

1. **Capture lane** — type meeting notes, simulate sample meeting, extract from transcript.
2. **Review lane** — extracted memory proposal cards with evidence/confidence/graph mutation.
3. **Write lane** — approve/reject controls, approve-all button, write status, and a Neo4j impact preview.

Update dashboard/header navigation so judges can discover L2 directly.

### Task 3: Add the L2 memory writer client component

**Objective:** Implement capture → extract → approve/reject → write status UX.

**Files:**
- Create: `components/l2-memory-writer.tsx`

**Implementation notes:**
- Use `"use client"`.
- Imports:
  - `useMemo`, `useState`
  - icons: `CheckCircle2`, `Database`, `FileText`, `Plus`, `RefreshCw`, `Sparkles`, `Trash2`, `XCircle`
  - types: `ClientContext`, `ExtractedMemory`, `TranscriptEvent`
  - UI: `Badge`, `EmptyState`, `Panel`
- State:
  - `events: TranscriptEvent[]`
  - `draft: string`
  - `extracted: ExtractedMemory[]`
  - `rejectedIds: string[]`
  - `approvedIds: string[]`
  - `savingIds: string[]`
  - `writeLog: Array<{ id: string; summary: string; saved: boolean; writeMode?: string; reason?: string }>`
  - `error: string | null`
  - `isExtracting: boolean`
- Add demo L2 statements:
  - Mr. Tan mentions wife expecting in September.
  - He wants estate planning advice before the baby arrives.
  - He asks Sarah to introduce Evelyn Ng.
  - Sarah promises to send an estate planning checklist.
- `addEvent(text)` appends `TranscriptEvent`.
- `extract()` POSTs to `/api/meetings/${meetingId}/extract` with `{ events }`, then stores `data.extracted`.
- `approveMemory(memory)` POSTs to `/api/memory/approve` with `{ memory }`, updates approved IDs and write log.
- `approveAll()` approves all visible unapproved cards sequentially.
- `rejectMemory(id)` adds to rejected IDs.
- UI columns:
  1. Transcript capture panel.
  2. Candidate memory cards panel.
  3. Write queue / Neo4j impact panel.

**Important behavior:**
- No auto-write on extraction.
- Every Neo4j write requires clicking Approve or Approve all.
- If Neo4j is not configured, the API returns `saved: false`; show that as `Demo fallback`, not as a silent success.

## Risks / Tradeoffs

- Current extraction is deterministic demo logic, not full GPT extraction. That is okay for hackathon reliability, but can be upgraded later.
- `/api/memory/approve` writes memory but not all action items. This PR should focus on memories first to keep scope tight.
- In demo mode or without Neo4j env, approve returns `saved: false`; UI must explain that Neo4j config is needed for actual persistence.

---

## Acceptance Criteria

- `/l2/[meetingId]` loads for the seeded meeting.
- User can simulate/type transcript statements.
- User can extract candidate memories.
- User can approve or reject each candidate.
- Approving calls `/api/memory/approve`.
- UI clearly shows whether Neo4j saved the memory or demo fallback occurred.
- Dashboard/header link to L2.
- `npm run check:l2`, `npm run lint`, `npm run build`, and `npm run check:mvp` pass.
